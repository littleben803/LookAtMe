# LookAtMe / 想恋爱 2.0 协作规范

## 用户与沟通

- 默认使用中文交流。
- 用户是资深 iOS 工程师，沟通应务实、简洁、专业。
- 对复杂技术问题，先给专业结论，再解释关键点。
- 不需要反复解释基础 iOS 概念，除非用户明确要求。

## 项目背景

「想恋爱」是一款 2015 年上线过的 iOS App，历史版本号为 1.0.0。老版本的核心能力是把手机变成可以输入文字、全屏展示、滚动播放的 LED 灯牌工具，典型场景包括演唱会应援、表白、生日祝福、接机、聚会整活等。

「想恋爱 2.0」不是简单复刻老版本，也不是临时 Demo。当前目标是从 0 到 1 重做一个高质量、完整、可上线、可长期维护的正式产品，保留老产品的情绪价值和核心场景，并用现代 iOS 设计、SwiftUI 架构、StoreKit 2 IAP 和高保真 UI 重新上线。

## 产品定位

- App 名称：想恋爱
- 一句话定位：把手机变成会发光的表白灯牌。
- App 类型：工具类 App。
- 主要语言：简体中文。
- 目标平台：iOS。
- 核心路径：打开 App -> 输入文字 -> 选择模板/样式/颜色/字体 -> 点击开始展示 -> 进入全屏 LED 霓虹展示页 -> 横屏举起手机使用。

## 硬性约束

- 免费用户必须能完整完成核心路径：输入文字 -> 选择基础模板/样式/颜色 -> 开始全屏展示。
- Pro 只能增强体验，不能阻断基础使用。
- Pro 重点解锁高级动效、高级模板、高级字体、无限收藏、自定义样式保存等。
- 首版本地优先：不做账号、不做服务端、不做云同步、不做社区、不做广告。
- 未经明确允许，不新增网络请求、账号系统、服务端、在线 AI、分析 SDK、广告追踪、StoreKit 以外的支付能力、CloudKit、iCloud 同步或敏感权限申请。
- 未经明确允许，不新增第三方依赖。
- 未经明确允许，不修改签名、Bundle ID、Team、Version、Build、App Store 配置。
- 不允许为了赶进度擅自砍掉已确定核心体验，也不允许把已确定功能做成不可用的假入口。
- 不要把实验功能接入正式默认路径，除非用户明确要求。
- 不要保存、打印或提交任何密钥、token、证书、Apple Developer 凭据、Hugging Face token、签名资产或私钥。
- 不要提交模型文件、二进制大文件、xcframework、签名文件、构建产物、本地环境文件或临时文件，除非用户明确允许。

## UI 风格

- 设计方向：方案 A，霓虹浪漫风。
- 整体视觉：深色、霓虹、浪漫、演唱会氛围。
- 主色：`#FF4DA6`。
- 背景色：`#0D0221` / `#160428`。
- 不允许做成普通系统表单风格。
- 不允许默认蓝色系统按钮破坏霓虹风格。
- 不允许页面间风格割裂。
- 不允许图标、文字、间距混乱。
- 现场使用优先：字体足够大、对比度足够强、横屏体验优秀、控制面板不遮挡内容、远距离可读。

## 技术方向

- SwiftUI。
- StoreKit 2。
- UserDefaults / AppStorage。
- 本地 JSON / Codable。
- 不接入第三方 SDK。
- 不接入广告 SDK。
- 不接入账号系统。
- 不接入服务端。

## 开发推进方式

- 不要一上来就直接写代码。
- 每次开发任务开始前，必须先明确：
  1. 产品目标
  2. 技术方案
  3. 任务边界
  4. 允许修改的文件
  5. 不允许修改的文件
  6. 验收标准
- 按阶段小步执行。
- 每完成一个阶段，先汇报结果，并等待用户确认后再进入下一阶段。
- 宁可边界清晰、修改小，也不要大范围重构。
- 结束任务前必须检查 `git status`，并说明是否存在异常文件。

## 阶段汇报格式

每次任务完成后，必须汇报：

- 本阶段是否完成
- 新增了哪些文件
- 修改了哪些文件
- 明确哪些重要区域未修改
- 运行了哪些验证命令
- Debug / Release 构建结果
- 是否有未解决问题
- 是否建议进入下一阶段
